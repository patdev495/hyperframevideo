# Premium News Karaoke Treatment PRD

## Problem Statement

The current **News-to-Video Pipeline** can produce a playable **Vertical Short Video**, but the resulting **HyperFrames Composition** looks like a basic text slide: each scene shows a title and the full narration body while the voiceover reads the same content. This is not visually competitive with professional short-form news videos.

The user wants a **Premium News Visual Treatment** that keeps the pipeline deterministic and local-first, but substantially improves perceived production quality through polished backgrounds, scene transitions, motion-designed text, and **Karaoke Caption** behavior. The caption must not show the full narration at once. It should reveal only a short phrase and highlight the active spoken word as the voiceover progresses.

The implementation must remain flexible enough to use approximate text-based timing now and precise word timestamps later, such as timestamps generated by Whisper or another forced-alignment provider when stronger GPU hardware is available.

## Solution

Add a new deterministic **Premium News Visual Treatment** that renders each scene with:

- Animated professional background layers generated in HTML/CSS/GSAP.
- Varied news-explainer layouts rather than a single centered text block.
- Smooth scene transitions.
- **Karaoke Caption** overlays that show a short phrase window and highlight the active word.
- A timing abstraction that supports multiple caption timing sources.

For the first implementation, caption timing will be approximate. The system will derive word-level timing from `narration_text` and the existing segment `duration_seconds` in `voiceover.json`. This makes the feature work without new GPU-heavy dependencies.

The caption timing model must be provider-neutral. A later implementation can add a Whisper-based provider that generates precise word timestamps from each voiceover WAV file without changing the composition renderer.

## User Stories

1. As a video producer, I want the generated video to look more polished than static text slides, so that the output feels closer to a professional social video.
2. As a video producer, I want motion-designed backgrounds, so that each scene has visual depth even without B-roll.
3. As a video producer, I want smooth transitions between scenes, so that the video feels continuous instead of a sequence of abrupt slides.
4. As a video producer, I want only the currently spoken phrase visible, so that the viewer is not overwhelmed by long narration text.
5. As a viewer, I want the active spoken word highlighted, so that I can follow the narration like karaoke captions.
6. As a viewer, I want the caption rhythm to feel synchronized with the voiceover, so that the video feels intentionally edited.
7. As a pipeline user, I want the first version to work without installing Whisper or a GPU dependency, so that I can use it on the current machine.
8. As a pipeline user, I want the caption timing format to support exact timestamps later, so that I can improve sync quality when I have better hardware.
9. As a developer, I want a clear caption timing provider boundary, so that approximate timing and Whisper timing can be tested independently.
10. As a developer, I want the **HyperFrames Composition** to remain deterministic, so that repeated renders from the same artifacts produce the same video.
11. As a developer, I want the generated HTML to follow HyperFrames requirements, so that preview, lint, inspect, and render remain reliable.
12. As a pipeline user, I want the existing `ai-modern` treatment to remain available, so that the new professional treatment does not break existing runs.
13. As a pipeline user, I want to select `premium-news` through the existing `Visual Treatment` header, so that the workflow does not need a new command.
14. As a pipeline user, I want downstream artifacts to fail clearly when required caption timing data is invalid, so that I can fix the run instead of getting a broken video.
15. As a developer, I want tests around the caption timing contract, so that future Whisper integration cannot silently break karaoke captions.
16. As a developer, I want tests around the generated composition, so that professional layout features do not regress back to static text.
17. As a future implementer, I want documented extension points for precise timestamp providers, so that Whisper integration can be added without redesigning the pipeline.

## Implementation Decisions

- Create a provider-neutral **Karaoke Caption** timing model with segment-relative word entries. Each entry should include text, start time, end time, and an order/index.
- Keep caption timing segment-relative in the data model. The composition generator can convert to absolute timeline time using the scene start time.
- Start with an approximate timing provider that derives word timings from `narration_text` and `duration_seconds`.
- Preserve a stable interface for future precise timing providers. The provider should accept the same narration/audio context and return the same caption timing model.
- Do not couple caption timing to VieNeu-specific internals. The installed VieNeu SDK currently exposes waveform generation and saving, but not public word timestamp APIs.
- Extend the voiceover/caption artifact contract in a way that supports optional caption timing without breaking existing `voiceover.json` consumers.
- Add `premium-news` as a new **Visual Treatment** rather than replacing `ai-modern`.
- Generate professional visuals with deterministic HTML/CSS/GSAP: layered backgrounds, accent highlights, motion panels, phrase windows, active word states, and transitions.
- Keep external B-roll, image generation, screenshots, and asset search out of scope for this PRD.
- Keep HyperFrames as the render dependency and do not fork or vendor the upstream HyperFrames project.
- Register all GSAP timelines on `window.__timelines` and avoid non-deterministic runtime behavior such as unseeded randomness.
- Keep the existing CLI flow: script approval, voiceover, storyboard, compose, render. The new treatment should be selected through the existing `Visual Treatment` artifact field.

## Testing Decisions

- Test external behavior, not internal implementation details.
- Add unit tests for approximate caption timing:
  - produces ordered words with monotonic start/end times;
  - covers the full segment duration;
  - handles punctuation and Vietnamese text;
  - keeps timing deterministic for the same input.
- Add schema/contract tests for caption timing serialization or manifest extension.
- Add composition generator tests that assert the `premium-news` output contains:
  - HyperFrames data timing attributes;
  - audio tracks;
  - karaoke caption data;
  - active-word animation hooks;
  - registered GSAP timeline;
  - treatment-specific background/transition layers.
- Add CLI-level tests where useful to ensure `Visual Treatment: premium-news` can flow from script/storyboard into composition.
- Preserve existing tests for `ai-modern`.

## Out of Scope

- Precise Whisper implementation.
- GPU setup for Whisper or forced alignment.
- B-roll search, image generation, logo extraction, screenshots, and asset downloads.
- Presenter avatar, lip sync, or talking-head video.
- Manual video editor UI.
- Music bed, audio mixing beyond current HyperFrames audio tracks.
- Replacing VieNeu-TTS.

## Further Notes

HyperFrames documentation confirms that compositions are plain HTML with `data-*` timing attributes and seekable timelines, and that captions synced to TTS narration are an expected composition pattern. This PRD keeps the repo aligned with that model while making the caption timing source replaceable.

The current VieNeu SDK in this workspace is `vieneu 3.0.4`. Local inspection shows public methods for `infer`, `infer_stream`, `infer_batch`, `save`, and voice helpers, but no public timestamp or word-alignment API. If a later VieNeu release exposes timestamps, it can be added as another precise caption timing provider using the same contract intended for Whisper.
